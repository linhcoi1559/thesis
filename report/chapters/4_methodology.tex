\chapter{METHODOLOGY}

This chapter presents the architecture and implementation methods used to build the \textit{\textbf{Smart Boarding House Management System}}. It explains the technology choices, system design, database structure, security mechanisms, business logic automation, and AI integration in detail.

\section{Tools and Techniques}

Choosing the right tools is very important to build a reliable rental management platform. The selection of the technology stack was driven not only by functional requirements but also by specific engineering goals: ensuring data consistency, enabling real-time communication, and supporting rapid development across a modular codebase. This section explains the technologies selected and the reasons for choosing them over common alternatives.

\subsection{Backend Services}
\begin{itemize}
    \item \textbf{NestJS (vs. Express):} While Express is a popular minimalist framework, \textbf{NestJS} is chosen because it provides a structured, modular architecture built on top of Express with built-in support for dependency injection, decorators, and guards. This structure is essential for organizing a large codebase into independent domain modules such as authentication, rooms, contracts, invoices, and AI. Without this structure, a plain Express application would quickly become difficult to maintain as the number of features grows.

    \item \textbf{PostgreSQL with Prisma ORM (vs. MongoDB):} \textbf{PostgreSQL} is chosen as the primary database because rental management data, particularly contracts and invoices, requires strict relational integrity and ACID-compliant transactions. For example, when a contract is created, the room status must be updated atomically in the same operation. A document database like \textbf{MongoDB} cannot enforce foreign key relationships or multi-table transactions as reliably. \textbf{Prisma ORM} is used on top of \textbf{PostgreSQL} because it provides type-safe query builders and auto-generated TypeScript types from the schema, which prevents type mismatch bugs at compile time and makes database migrations traceable.

    \item \textbf{Socket.IO (vs. HTTP Polling):} Instead of having the frontend repeatedly send HTTP requests to check for new notifications, \textbf{Socket.IO} is used to maintain a persistent WebSocket connection between the server and each logged-in client. This allows the backend to push events such as invoice payment confirmations and new incident reports to the landlord's browser instantly, without any delay or wasted network traffic from polling.
\end{itemize}

\subsection{Frontend Portal}
\begin{itemize}
    \item \textbf{Next.js 14 App Router (vs. React SPA):} \textbf{Next.js} is chosen instead of a standard single-page application because it supports server-side rendering. This is important for the public landing page, where room listings must be visible to search engines to attract potential tenants. The App Router also provides a clean file-based routing system that naturally separates the three access contexts: public pages, the landlord administration dashboard, and the tenant portal.

    \item \textbf{Zustand (vs. Redux):} For global state management, \textbf{Zustand} is chosen over \textbf{Redux} because it requires significantly less boilerplate code. Redux requires defining actions, reducers, and selectors separately, which adds unnecessary complexity for managing simple states such as the logged-in user session and UI preferences. Zustand achieves the same result with a simple hook-based store, making the codebase easier to read and maintain.

    \item \textbf{Axios with Interceptors:} For API requests, \textbf{Axios} is used with custom request interceptors. These interceptors automatically attach the \textbf{JWT} Bearer token and the landlord identification header to every outgoing request, so individual components do not need to manage authentication headers manually. A response interceptor also handles token expiry errors globally.
\end{itemize}

\subsection{Supporting Services and Integrations}
\begin{itemize}
    \item \textbf{Cloudinary (vs. Local File Storage):} Room images are stored on \textbf{Cloudinary} instead of on the local server disk. This avoids filling up the server's disk space and allows \textbf{Cloudinary} to automatically optimize and resize images for faster loading on public devices. The backend stores only the secure image URL in \textbf{PostgreSQL}, keeping the database lightweight.

    \item \textbf{Google Gemini (vs. OpenAI GPT):} The \textbf{Gemini 2.0 Flash Lite} model is chosen for the AI assistant features because it offers a very fast response time and a generous free-usage tier, which keeps operational costs low for a student project. It is accurate enough for the three required tasks: generating formal Vietnamese announcements, answering room availability questions, and answering tenant billing questions from structured database context.

    \item \textbf{VietQR API:} The \textbf{VietQR} image API is called dynamically when an invoice PDF is generated. It produces a QR code image pre-filled with the landlord's bank account number and the exact invoice amount. Tenants can scan this code with any Vietnamese banking application to complete the transfer instantly, without needing to type account numbers manually.
\end{itemize}

\subsection{Infrastructure and Deployment}
\begin{itemize}
    \item \textbf{Neon (Serverless PostgreSQL):} The \textbf{PostgreSQL} database is hosted on \textbf{Neon}, a serverless cloud database provider. Neon automatically scales compute resources based on demand and provides connection pooling, which is important because serverless backends on \textbf{Render} can spin up many instances simultaneously.

    \item \textbf{Render (Backend Hosting):} The \textbf{NestJS} backend is deployed as a Web Service on \textbf{Render}. \textbf{Render} detects the backend directory from the \textbf{GitHub} repository, runs the build command automatically, and connects to the \textbf{Neon} database through environment variables.

    \item \textbf{Vercel (Frontend Hosting):} The \textbf{Next.js} frontend is deployed on \textbf{Vercel} because \textbf{Vercel} is built and optimized specifically for \textbf{Next.js} applications. It handles server-side rendering infrastructure automatically and provides a global CDN for fast page loads. Both \textbf{Render} and \textbf{Vercel} are connected to the \textbf{GitHub} repository, enabling automatic \textbf{CI/CD}: any code pushed to the main branch is built and deployed to production without manual intervention.
\end{itemize}


\section{System Architecture}

This section describes how the different parts of the Smart Boarding House Management System are organized and how they connect with each other. The system is designed using a clean, layered structure to separate the user interface, business logic, and data persistence concerns.

\subsection{Overall System Structure}
The system follows a \textbf{client-server} architecture. The frontend built with \textbf{Next.js} acts as the client, and the backend built with \textbf{NestJS} acts as the server. As shown in Figure \ref{fig:system_architecture}, the client and server communicate using HTTP \textbf{REST APIs} for standard data requests and \textbf{Socket.IO} WebSockets for real-time event notifications. The backend server connects directly to the \textbf{PostgreSQL} database on \textbf{Neon} to read and store all rental information. It also communicates with three external services: \textbf{Cloudinary} for image storage, \textbf{Google Gemini} for AI features, and the \textbf{VietQR API} for payment QR code generation.

    \centering
    \includegraphics[width=1.0\textwidth]{SystemArchitecture.png}
    \caption{Overall System Architecture}
    \label{fig:system_architecture}
\end{figure}

\textbf{Design Rationale:} A single backend service rather than a microservices architecture is chosen deliberately. Because the project is built and maintained by a single developer within a limited timeframe, a monolithic \textbf{NestJS} application organized into domain modules provides the right balance between structure and simplicity. Each domain module (rooms, contracts, invoices, AI, etc.) is self-contained and could be extracted into a separate service in a future version without significant refactoring.

\subsection{Layered Design}
To make the codebase easy to maintain and extend, both the \textbf{backend} and \textbf{frontend} are organized into three distinct layers: the \textbf{presentation layer}, the \textbf{application layer}, and the \textbf{infrastructure layer}. This separation ensures that changes in one layer do not directly affect the others.

The \textbf{presentation layer} is the top layer. In the \textbf{backend}, it contains \textbf{NestJS} HTTP controllers and Data Transfer Objects (DTOs) that receive incoming requests, validate their shape, and send formatted responses. In the \textbf{frontend}, it contains the \textbf{Next.js} pages and \textbf{React} components that render the UI. This layer does not contain any business rules; its sole responsibility is to pass validated data to the next layer.

The \textbf{application layer} contains the core business logic. For example, when a landlord creates a contract, it is this layer that checks whether the selected room is vacant, executes the database transaction, and updates the room status. It contains service classes for authentication, room management, contract lifecycles, invoice creation, and notification dispatch. This layer is completely independent of the specific database library or HTTP framework being used.

The \textbf{infrastructure layer} is the bottom layer. It handles all communication with external systems. In the \textbf{backend}, this includes \textbf{Prisma} repository classes for executing \textbf{PostgreSQL} queries, a \textbf{Cloudinary} upload service, and the \textbf{Google Generative AI} client. This layered design ensures that, for example, switching from \textbf{Prisma} to another ORM would only require changes in the infrastructure layer without touching the business logic above it.


\section{Database Design}

The database is the foundation of the platform, storing all rental information in a structured and consistent way. \textbf{PostgreSQL} is used as the database and \textbf{Prisma ORM} manages the connection and schema. This section explains the entity relationships, the purpose of each table, and how the schema enforces multi-tenant data privacy.

\subsection{Entity-Relationship Diagram}
To understand how the data is structured, an \textbf{Entity-Relationship Diagram (ERD)} is created. As shown in Figure \ref{fig:erd_diagram}, the database contains eight main tables that represent the core domain objects of the boarding house system: \textbf{User}, \textbf{Landlord}, \textbf{Room}, \textbf{Contract}, \textbf{Invoice}, \textbf{RentRequest}, \textbf{Incident}, and \textbf{Violation}.

\subsection{Core Database Tables}
The \textbf{User} table is the central identity record for all accounts in the system. Each user record stores an email, a bcrypt-hashed password, a display name, a phone number, an account status (\textbf{PENDING}, \textbf{APPROVED}, \textbf{REJECTED}, or \textbf{BANNED}), and a role (\textbf{ADMIN}, \textbf{LANDLORD}, or \textbf{TENANT}). If the account belongs to a tenant, the record also stores a \texttt{landlordId} foreign key that connects the tenant to their specific property owner. New user registrations are saved with \textbf{PENDING} status by default, so landlords must explicitly approve tenants before they can log in.

The \textbf{Landlord} table stores the business profile of each property owner. It records the landlord's name, phone, address, and bank account details (bank name, account holder name, and account number). These bank details are critical because they are fetched automatically every time an invoice PDF is generated to produce the \textbf{VietQR} payment code.

The \textbf{Room} table represents a rentable unit. Each room record stores a room number, floor, description, a decimal rental price, a status field (\textbf{VACANT}, \textbf{OCCUPIED}, or \textbf{MAINTENANCE}), and an array of Cloudinary image URLs. Every room is associated with a landlord through a \texttt{landlordId} foreign key with cascade deletion, meaning all room records are automatically removed if the landlord account is deleted.

The \textbf{Contract} table represents the formal lease agreement between a landlord, a tenant, and a room. It stores the contract number, start date, end date, deposit amount, monthly rental price, electricity unit price, water unit price, member count, and a status field (\textbf{PENDING}, \textbf{ACTIVE}, \textbf{EXPIRED}, or \textbf{TERMINATED}). The contract also stores a boolean \texttt{autoPayEnabled} flag that controls whether the monthly cron job should automatically generate an invoice for this contract.

The \textbf{Invoice} table stores individual monthly billing records. Each invoice is linked to a contract and records the total amount due, the breakdown of electricity cost, water cost, other costs, the due date, the actual payment date, and a status (\textbf{UNPAID}, \textbf{PAID}, \textbf{OVERDUE}, or \textbf{CANCELLED}). A unique invoice number is generated for each record to allow tenants and landlords to identify payments easily.

The \textbf{RentRequest}, \textbf{Incident}, and \textbf{Violation} tables support the operational side of the platform. \textbf{RentRequest} stores inquiries from public visitors who are interested in a room. \textbf{Incident} stores maintenance or repair requests submitted by tenants, each with a priority level (\textbf{LOW}, \textbf{MEDIUM}, or \textbf{HIGH}) and a resolution status. \textbf{Violation} stores rule infractions recorded by the landlord against specific tenants.

\subsection{Multi-Tenant Data Privacy}
Because multiple landlords share the same \textbf{PostgreSQL} database instance, preventing cross-tenant data leakage is a critical design concern. To solve this, every business table — \textbf{Room}, \textbf{Contract}, \textbf{Invoice}, \textbf{RentRequest}, \textbf{Incident}, and \textbf{Violation} — includes a \texttt{landlordId} column with a database-level index. When a landlord logs in, their \texttt{landlordId} is embedded in the \textbf{JWT} token. Every service method that queries these tables automatically appends a \texttt{WHERE landlordId = ?} filter using the value extracted from the authenticated request object. This means that even if a landlord attempts to request another landlord's invoice by guessing its UUID, the query returns no results because the \texttt{landlordId} filter does not match.


\section{Authentication and Authorization}

Security is a critical part of the platform. The system must ensure that only registered and approved users can access private pages, and that landlords can only access their own rental data. To achieve this, the backend implements token-based authentication and a layered role and ownership authorization system.

\subsection{User Authentication with JWT}

The authentication workflow uses \textbf{JSON Web Tokens (JWT)} to issue and verify user identities. The detailed execution steps are as follows:

\begin{enumerate}
    \item The user submits their email and password to the \texttt{POST /auth/login} endpoint.
    \item The \textbf{AuthService} queries the \textbf{User} table for a record with the matching email. If no record is found, an unauthorized exception is thrown immediately.
    \item The service calls \textbf{bcrypt.compare} to verify the submitted password against the stored hash. If the comparison fails, an unauthorized exception is thrown.
    \item If the credentials are valid, the service calls \textbf{JwtService.sign} to generate a signed \textbf{JWT} token. The token payload includes the user's UUID (\texttt{sub}), email, role, and \texttt{landlordId}, as described in Table~\ref{tab:jwt_payload}.
    \item The signed token is returned to the client. The frontend stores it in memory and attaches it as a \texttt{Bearer} token to the \texttt{Authorization} header of every subsequent API request using an \textbf{Axios} interceptor.
\end{enumerate}

\textbf{Implementation Highlights:}
\begin{itemize}
    \item \textbf{Passport JWT Strategy:} Token validation is implemented as a \textbf{NestJS Passport} strategy. When a protected request arrives, the strategy extracts the token from the \texttt{Authorization} header, verifies its cryptographic signature using the server's secret key, and attaches the decoded payload to the request object as \texttt{req.user}. This decoded object is then available to all downstream guards and service methods without any additional database queries.
    \item \textbf{Password Hashing:} All passwords are hashed with \textbf{bcrypt} using a salt round of 10 before being saved to the database. Plain-text passwords are never stored or logged at any point.
    \item \textbf{Axios Interceptors on the Frontend:} The \textbf{Next.js} frontend uses a custom \textbf{Axios} instance with a request interceptor that reads the \textbf{JWT} token from the \textbf{Zustand} auth store and attaches it to every outgoing request automatically. This ensures that individual components never need to manage authentication headers directly.
\end{itemize}

\subsection{API Protection with Auth Guards}

To protect individual API endpoints, the backend implements two custom \textbf{NestJS} guards that are applied to controllers using decorators. As shown in Figure \ref{fig:auth_guard_flowchart}, every protected request passes through both guards before reaching the route handler.

\textbf{Implementation Highlights:}
\begin{itemize}
    \item \textbf{JwtAuthGuard:} This guard is the first checkpoint. It calls the \textbf{Passport JWT Strategy} to extract and verify the token. If the token is missing, malformed, or expired, the guard immediately throws an \texttt{UnauthorizedException} and the request never reaches the controller. On success, it attaches the decoded user payload to the request object.
    \item \textbf{RolesGuard:} This guard is applied after \textbf{JwtAuthGuard} on routes that require a specific role. It uses the \textbf{NestJS Reflector} to read the \texttt{@Roles()} decorator metadata on the current route handler. It then compares the \texttt{role} field from the decoded \textbf{JWT} payload against the required roles. If the user's role does not match (for example, a tenant trying to access a landlord-only route), the guard throws a \texttt{ForbiddenException}.
    \item \textbf{Public Route Bypass:} Routes that must be accessible without authentication (such as \texttt{GET /rooms/public}, \texttt{POST /auth/login}, and \texttt{POST /ai/public-chat}) are decorated with a custom \texttt{@Public()} metadata marker. The \textbf{JwtAuthGuard} checks for this marker using the \textbf{Reflector} and skips token validation for those routes entirely.
\end{itemize}

\subsection{Multi-Tenant Authorization Filter}

Apart from roles, the system enforces data ownership using the \texttt{landlordId} embedded in the \textbf{JWT} token. Because multiple landlords share the same database, the backend must prevent one landlord from reading or modifying another landlord's records.

\textbf{Execution Flow:}
\begin{enumerate}
    \item When a landlord logs in, the \textbf{AuthService} fetches the landlord's \texttt{landlordId} from the \textbf{Landlord} table and embeds it in the \textbf{JWT} payload.
    \item On every subsequent protected request, the \textbf{JwtAuthGuard} decodes the token and attaches the full payload — including \texttt{landlordId} — to \texttt{req.user}.
    \item The service method receives the \texttt{landlordId} from \texttt{req.user} and passes it as a mandatory filter in every \textbf{Prisma} query: \texttt{where: \{ landlordId: req.user.landlordId \}}.
    \item The database returns only records that belong to that specific landlord, making it structurally impossible for one landlord to access another's data through the API.
\end{enumerate}

\textbf{Implementation Highlights:}
\begin{itemize}
    \item \textbf{No Separate Middleware:} The multi-tenant filter is not implemented as a separate middleware layer. Instead, the \texttt{landlordId} is injected directly into each service method call. This keeps the authorization logic co-located with the business logic and avoids any risk of the filter being accidentally bypassed by a new route.
    \item \textbf{Database Indexes:} All business tables have a database index on the \texttt{landlordId} column, defined in the \textbf{Prisma} schema using \texttt{@@index([landlordId])}. This ensures that the landlord-scoped filter queries remain fast even as the number of records grows.
\end{itemize}

\subsection{Room and Tenant Management}

Room management is the foundation of the platform. Before contracts or invoices can be created, the landlord must register rooms and approve tenant accounts. The room creation workflow validates all required fields on the backend using \textbf{NestJS} class-validator DTOs before saving to the database.

\textbf{Execution Workflow — Room Image Upload:}
\begin{enumerate}
    \item The landlord selects an image file in the dashboard and submits it to \texttt{POST /rooms/:id/images}.
    \item The \textbf{NestJS} controller receives the file via \textbf{Multer} middleware.
    \item The \textbf{RoomService} calls the \textbf{CloudinaryService}, which uploads the binary file to the \textbf{Cloudinary} cloud using the official SDK.
    \item \textbf{Cloudinary} returns a secure HTTPS URL for the uploaded image.
    \item The service appends this URL to the room's \texttt{imageUrls} array in \textbf{PostgreSQL} using a \textbf{Prisma} update.
    \item The updated room record is returned to the frontend, which immediately reflects the new image in the room card.
\end{enumerate}

\textbf{Implementation Highlights:}
\begin{itemize}
    \item \textbf{Public Room Listing:} The \texttt{GET /rooms/public} endpoint is decorated with \texttt{@Public()} and requires no authentication. It queries only rooms with \texttt{status: VACANT} across all landlords to populate the public landing page. This is the only cross-tenant query in the system and is intentionally scoped to vacant rooms only to prevent privacy leakage.
    \item \textbf{Tenant Account Status:} New users who register with the \textbf{TENANT} role receive \texttt{PENDING} status. The landlord must explicitly call \texttt{PATCH /tenants/:id/status} to set the status to \textbf{APPROVED} before the tenant can log in and access the portal. This prevents unauthorized users from accessing private rental data.
\end{itemize}

\section{Automation and Business Logic}

This section describes the technical implementation of the key business processes in the backend. To ensure data consistency and provide a reliable user experience, the system automates several workflows using database transactions, scheduled background tasks, and real-time WebSocket messaging.

\subsection{Rental Contract Creation}

Creating a rental contract requires updating multiple database records atomically. The most critical requirement is that a room's status must change from \textbf{VACANT} to \textbf{OCCUPIED} at the exact same moment the contract record is saved. If these two operations are not atomic, a system crash between them could leave the database in an inconsistent state where a room is occupied but has no active contract, or a contract exists for a room that is still marked vacant.

\textbf{Execution Workflow:}
\begin{enumerate}
    \item The landlord submits contract details (tenant, room, dates, deposit, and rental price) to \texttt{POST /contracts}.
    \item The \textbf{ContractService} first validates that the selected room's \texttt{landlordId} matches the requesting landlord's ID, preventing a landlord from creating a contract for another landlord's room.
    \item The service queries the \textbf{Room} table to confirm the room's current status is \textbf{VACANT}. If the room is already occupied, it throws a conflict exception and stops execution.
    \item The service opens a \textbf{Prisma} interactive transaction. Inside the transaction, it executes two operations: (a) creating the new \textbf{Contract} record, and (b) updating the \textbf{Room} status to \textbf{OCCUPIED}.
    \item If both operations succeed, the transaction is committed and the new contract is returned. If any operation throws an error, \textbf{Prisma} automatically rolls back both changes, leaving the database in its original clean state.
\end{enumerate}

    \centering

    \caption{Activity Diagram - Contract Creation Process}
    \label{fig:activity_create_contract}

\textbf{Implementation Highlights:}
\begin{itemize}
    \item \textbf{Prisma Transactions:} The atomic update is implemented using \textbf{Prisma's} \texttt{\$transaction()} API. This API accepts an array of Prisma operations and executes them inside a single database transaction. This is the most reliable way to ensure that the contract and room status are always synchronized.
    \item \textbf{Contract Expiration Cron Job:} A separate \textbf{NestJS} scheduled task runs every day at midnight using the \texttt{@Cron()} decorator. It queries all contracts whose \texttt{endDate} has passed and whose status is still \textbf{ACTIVE}. For each expired contract, it updates the contract status to \textbf{EXPIRED} and updates the associated room status back to \textbf{VACANT} inside another Prisma transaction.
    \item \textbf{Startup Consistency Check:} When the backend application starts, a one-time bootstrap task runs to detect any rooms that are marked \textbf{OCCUPIED} but have no active contract (which could occur after an unexpected server crash). These rooms are automatically set back to \textbf{VACANT} to restore database consistency before the system begins serving requests.
\end{itemize}

\subsection{Invoice Payment and Verification}

The monthly billing cycle is managed through two mechanisms: an automated cron job that creates invoices, and a structured tenant-initiated payment flow.

\textbf{Execution Workflow:}
\begin{enumerate}
    \item At the beginning of each month, a \textbf{NestJS} \texttt{@Cron()} task runs automatically. It queries all \textbf{Contract} records with status \textbf{ACTIVE} and \texttt{autoPayEnabled = true} and creates a new \textbf{Invoice} record for each one, pre-filling the rental price and setting the due date to the end of the current month.
    \item The tenant logs in to the portal and navigates to the invoice list. The frontend calls \texttt{GET /invoices} to fetch all invoices linked to the tenant's active contract.
    \item The tenant clicks \textit{Pay} on an unpaid invoice. The frontend calls \texttt{GET /pdf/invoice/:id}, which triggers the backend to fetch the landlord's bank account details, call the \textbf{VietQR API} with the amount and account details, receive a QR code image in return, embed it into a \textbf{PDF} document, and return the file to the browser.
    \item The tenant scans the \textbf{VietQR} code with their banking application and completes the bank transfer.
    \item After transferring, the tenant clicks the \textit{Already Transferred} button. The frontend calls \texttt{POST /invoices/:id/pay}.
    \item The \textbf{InvoiceService} updates the invoice status to \textbf{PAID} and records the current timestamp in the \texttt{paidDate} field.
    \item Simultaneously, the backend emits a \textbf{Socket.IO} event to the landlord's WebSocket room, pushing a real-time notification to the landlord's dashboard so they are immediately informed of the payment.
\end{enumerate}

\textbf{Implementation Highlights:}
\begin{itemize}
    \item \textbf{VietQR Code Generation:} The \textbf{VietQR} image API is called at PDF generation time with the landlord's bank name, account number, account holder name, and the exact invoice amount as query parameters. The API returns a PNG image of the QR code, which is then embedded into the PDF using the \textbf{PDFKit} library. The transfer memo is also pre-filled automatically with the tenant's name and the invoice month, so the landlord can identify payments in their bank statement without any manual cross-referencing.
    \item \textbf{Socket.IO Notification Gateway:} The \textbf{NotificationGateway} is a \textbf{NestJS} WebSocket gateway decorated with \texttt{@WebSocketGateway()}. When a client connects, the gateway authenticates the connection by verifying the \textbf{JWT} token sent in the WebSocket handshake. It then adds the client's socket to a named room identified by the landlord's ID (e.g., \texttt{landlord-abc123}). When a payment event occurs, the backend emits directly to this room, ensuring the notification reaches only the correct landlord.
    \item \textbf{Role-Gated Pay Endpoint:} The \texttt{POST /invoices/:id/pay} endpoint is restricted to the \textbf{TENANT} role using the \textbf{RolesGuard}. This prevents a landlord from accidentally marking their own invoices as paid from the admin panel. Landlords must use the \texttt{PATCH /invoices/:id/status} endpoint to update invoice status manually after verifying the bank transfer.
\end{itemize}

\subsection{Tenant Incident Handling}

Managing room repair requests requires tight coordination between tenants and landlords. The incident workflow is designed so that the landlord is notified instantly when a new issue is reported, and the tenant is notified in real time when their issue is resolved.

\textbf{Execution Workflow:}
\begin{enumerate}
    \item The tenant opens the incident report form in the portal and submits a title, description, and priority level (\textbf{LOW}, \textbf{MEDIUM}, or \textbf{HIGH}) to \texttt{POST /incidents}.
    \item The \textbf{IncidentService} saves the new incident record to the database with status \textbf{PENDING} and associates it with the tenant's \texttt{landlordId}.
    \item Immediately after saving, the service calls the \textbf{NotificationGateway} to emit a WebSocket event to the landlord's room, notifying the dashboard of the new incident in real time.
    \item The landlord reviews the incident in the dashboard and calls \texttt{PATCH /incidents/:id/status} to update the status to \textbf{IN\_PROGRESS}, indicating that a technician has been assigned.
    \item Once the issue is resolved, the landlord updates the status to \textbf{RESOLVED}. The backend emits another WebSocket event, this time to the tenant's portal session, informing them that their request has been closed.
\end{enumerate}

\textbf{Implementation Highlights:}
\begin{itemize}
    \item \textbf{Priority Levels:} The \texttt{priority} field accepts three enum values: \textbf{LOW}, \textbf{MEDIUM}, and \textbf{HIGH}. This allows landlords to sort and filter incidents by urgency in the dashboard, so high-priority repairs (such as a water leak) are addressed before low-priority ones (such as a broken light switch).
    \item \textbf{Bidirectional Real-Time Notification:} Unlike invoice notifications, which only go from tenant to landlord, incident status updates flow in both directions. When the landlord updates the status, the backend checks whether the new status is \textbf{RESOLVED} and, if so, emits a notification event to the tenant's active WebSocket session as well.
\end{itemize}

\section{AI Integration}

The platform integrates \textbf{Google Gemini} to automate administrative writing tasks and to provide intelligent self-service assistants for both public visitors and registered tenants. The key design principle is that the AI model is never asked to guess or invent information — it is always given real, live data from the database as structured context before answering any question.

\subsection{AI Assistant with Google Gemini}

The AI service is built using the official \textbf{Google Generative AI} Node.js library targeting the \textbf{Gemini 2.0 Flash Lite} model. This model is chosen because it has a very fast response time, making it suitable for real-time chat interactions. The service implements three distinct operating modes, each targeting a different user role and injecting different database context, as summarized in Table~\ref{tab:ai_modes}.

    \item The \textbf{AiService} extracts the tenant's \texttt{userId} from the \textbf{JWT} token in the request.

    \item The service formats this data into a structured plain-text system prompt block describing the tenant's rental situation.
    \item It combines the system prompt with the tenant's chat message and sends the assembled prompt to the \textbf{Gemini} model via the \textbf{generateContent} API call.

\textbf{Implementation Highlights:}
\begin{itemize}
    \item \textbf{Context Injection to Prevent Hallucination:} The most important design decision in the AI integration is that all three assistant modes inject real database records as plain-text context into the system prompt before the user's question is passed to the model. This prevents the model from inventing room prices, contract terms, or billing amounts. For the RentBot public assistant, the backend fetches the current list of vacant rooms and formats it as a structured list. For the tenant assistant, it fetches the specific tenant's contract and invoice data. The model is instructed in the system prompt to answer only based on the provided context and to decline questions outside of its scope.
    \item \textbf{Stateless Single-Turn Design:} The current implementation uses a stateless single-turn prompt model. Each API call sends the full context plus the current question, without maintaining a multi-turn conversation history. This keeps the implementation simple and avoids the cost of storing and retrieving conversation state between requests. It is sufficient for the current use cases, where each question is typically self-contained.
    \item \textbf{Notice Generator Mode:} For landlord notice generation, no database context is injected. The landlord types a short informal instruction (for example, \textit{"remind tenants to pay by the 5th"}), and the system prompt instructs the model to rewrite it as a formal, polite Vietnamese boarding house announcement. This automates a task that previously required landlords to write formal notices manually.
\end{itemize}
